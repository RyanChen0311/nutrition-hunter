# SUB.md — 食物推薦子系統

> **角色**：接收來自 PLAN.md 的營養缺口向量，透過兩階段貪婪演算法從食物資料庫中挑選並組合出當日飲食推薦清單。

---

## 一、模組職責

```
gap{}（來自 PLAN.md 或使用者已攝取後的剩餘缺口）
                ↓
         SUB.md（本模組）
    Phase 1：主力食物選擇
    Phase 2：互補食物迭代
                ↓
    推薦清單 + 達成率（足夠／略低／不足）+ 提示
                ↓
        前端六角形視覺化 + 食物卡片
```

---

## 二、輸入規格

### 2.1 缺口向量（gap）

依 TEMP.md 約定，接收以下五個欄位：

| 欄位 | 型別 | 單位 | 說明 |
|------|------|------|------|
| carb_g | float | g | 碳水化合物缺口 |
| prot_g | float | g | 蛋白質缺口 |
| fat_g | float | g | 脂肪缺口 |
| fiber_g | float | g | 膳食纖維缺口 |
| micro_score | float | 0–100 | 維生素＋礦物質綜合缺口 |

若使用者今日尚未進食，gap = NutritionTarget，直接帶入。

### 2.2 系統參數（預設值）

| 參數 | 預設值 | 說明 |
|------|--------|------|
| 推薦食物種數上限 | 8 種 | 避免清單過長 |
| 初始估算份量 | 150 g | Phase 1/2 得分計算基準（注意：前端加入食物預設 100g） |
| 最小改善幅度 | 1% | Phase 2 終止條件 |
| 達成閾值 | 90% | 判斷「足夠」的門檻 |

> **注意**：推薦演算法內部以 150g 為估算基準計算得分，但前端「加入今日」的攝取量以 100g 計算，兩者為不同用途。

## 三、食物資料庫結構（`food_db.json`）

基礎單位：每 100 公克。資料來源參考 USDA FoodData Central 與 WHO 公開資料。

### 3.1 欄位定義

| 欄位 | 型別 | 說明 |
|------|------|------|
| id | int | 唯一識別碼（新增食物時自動遞增） |
| name | str | 食物名稱（市售品名風格，例：黃金薯條、北海道納豆） |
| category | str | 主食／肉類／蔬菜／水果／乳製品／堅果／豆類／蛋類 |
| carb | float | 碳水化合物（g/100g） |
| protein | float | 蛋白質（g/100g） |
| fat | float | 脂肪（g/100g） |
| fiber | float | 膳食纖維（g/100g） |
| micro_score | float | 維生素＋礦物質綜合得分（0–100，已正規化） |
| sodium | float | 鈉（mg/100g，用於未來鈉超標警示） |

### 3.2 食物命名原則

食物名稱採**市售品名風格**，而非原型食材名，使使用者更容易辨識：

| 原型名 | 資料庫名稱 |
|--------|----------|
| 地瓜 | 黃金薯條 |
| 燕麥 | 桂格即食燕麥 |
| 納豆 | 北海道納豆 |
| 雞胸肉 | 舒胸嫩雞排 |
| 鮭魚 | 挪威煙燻鮭魚 |

### 3.3 micro_score 正規化方式

```
vit_score  = Σ_k ( food_vit_k / daily_vit_k ) × vit_weight_k × 100
min_score  = Σ_k ( food_min_k / daily_min_k ) × min_weight_k × 100
micro_score = (vit_score + min_score) / 2
```

維生素權重：A 20%、C 25%、D 20%、B12 20%、葉酸 15%
礦物質權重：鈣 25%、鐵 25%、鋅 20%、鎂 15%、鉀 15%

### 3.5 使用者自訂食物

透過 `POST /api/foods` 可新增食物，持久化寫入 `food_db.json`。前端提供表單填寫 8 個欄位（名稱、分類、碳水、蛋白質、脂肪、纖維、微量評分、鈉），新增後即時顯示於食物資料庫卡片軸。

## 四、核心公式

### 4.1 各食物的填補貢獻

```
contrib_ij = n_ij × x_i / 100    # n_ij：每100g含量，x_i：選用公克數（預設150g）
```

### 4.2 各營養素達成率

```
R_j = Σ_i contrib_ij / gap_j     # 理想值 = 1.0
```

### 4.3 Phase 1 & Phase 2 得分

```
score_i = Σ_j  w_j × min( n_ij × 1.5 / gap_j,  1.0 )
```

x_default = 1.5 份（150g）為估算基準，w_j 為各營養素權重（預設均等 0.2）。Phase 2 以更新後的 gap_j（扣除已選食物貢獻）重新計算，自然產生互補效果。

---

## 五、兩階段貪婪演算法

### Phase 1：主力食物

1. 對資料庫所有食物計算 score_i
2. 選出得分最高者加入推薦清單
3. 更新缺口：`gap_j ← max(0, gap_j - contrib_ij)`

### Phase 2：互補迭代

重複以下步驟直到終止：

1. 以更新後 gap_j 重新對未選食物計算 score_i
2. 選出得分最高者加入清單，更新缺口
3. 檢查終止條件

**終止條件（任一滿足即停止）：**

```
A：所有 R_j 達到「足夠」（≥ 90%）
B：已選食物種數 = 8
C：剩餘最大缺口改善幅度 < 1%
```

> Phase 3（份量微調 ±0.5份迭代）於設計階段評估後移除。理由：日常飲食本就存在誤差，過度精確的微調對普通使用者沒有實際意義。

## 六、達成率語意分級

| 等級 | 達成率 | 顯示文字 | 六角形顏色 |
|------|--------|---------|-----------|
| 足夠 | ≥ 90% | ✓ 已達標 | 綠色 |
| 略低 | 70–89% | 略低，可再補充 | 橘色 |
| 不足 | < 70% | 不足，建議優先補充 | 紅色 |

**所有達標時：** 前端偵測到所有 gap ≤ 0.1 即禁用「取得推薦」按鈕，改顯示「已達標」，推薦面板顯示「所有營養素已達標！不需要額外補充食物。」

## 七、輸出結構（`RecommendResult`）

```python
RecommendResult = {
    "foods": [
        {
            "name":     str,
            "category": str,
            "amount_g": float,    # 推薦份量（g，演算法估算 150g）
            "contrib":  {
                "carb_g":      float,
                "prot_g":      float,
                "fat_g":       float,
                "fiber_g":     float,
                "micro_score": float,
            }
        }
    ],
    "coverage": {
        "carb":  "足夠" | "略低" | "不足",
        "prot":  "足夠" | "略低" | "不足",
        "fat":   "足夠" | "略低" | "不足",
        "fiber": "足夠" | "略低" | "不足",
        "micro": "足夠" | "略低" | "不足",
    },
    "tips":   [ str ],
    "ratios": {
        "carb":  float,   # 達成率（0.0 – 1.5+）
        "prot":  float,
        "fat":   float,
        "fiber": float,
        "micro": float,
    }
}
```
