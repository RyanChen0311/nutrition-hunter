## 階段一：六大營養素基礎知識確立

### 1.1 WHO 六大營養素定義

確認世界衛生組織定義的六大營養素：碳水化合物、蛋白質、脂肪、維生素、礦物質，以及第六項的確認。

> **設計決策**：若排除水，第六項為**膳食纖維**。雖然膳食纖維屬於碳水化合物的子類，但因其對消化健康、血糖調節的重要性，WHO 及多數營養學機構將其單獨列出。本系統採用「不含水、含膳食纖維」的六項分類。

## 階段二：TDEE 公式推導

### 2.1 問題設定

目標：建立能根據個人變數（體重、身高、年齡、性別、活動量）計算每日總消耗熱量的公式。

### 2.2 假設條件

- 每日睡眠固定 8 小時（不可運動狀態）
- 其餘 16 小時為清醒狀態（可能運動或靜態）
- 拆分為：睡眠消耗 + 清醒靜態消耗 + 運動消耗

### 2.3 BMR — Mifflin-St Jeor 公式（1990）

```
男性：BMR = 10W + 6.25H - 5A + 5
女性：BMR = 10W + 6.25H - 5A - 161
```

### 2.4 TDEE 整合公式（最終版）

```
TDEE = BMR × (23.6 - T_ex) / 24 + Σ(MET_i × W × T_ex_i)
```

推導說明：睡眠項 0.95 × BMR × 8/24 = 7.6/24；靜態項 BMR × (16 - T_ex)/24；合併係數 7.6 + 16 = 23.6。

> **驗證**：T_ex = 0 時，TDEE ≈ 0.983 × BMR，符合純靜態生活的實際消耗。

## 階段三：六大營養素目標量公式推導

依 WHO 建議：
- 碳水 55%：`C_carb = 0.55 × TDEE / 4`
- 蛋白質雙方法取最大值（熱量比例 + 體重基準）
- 脂肪 30%：`C_fat = 0.30 × TDEE / 9`
- 膳食纖維：每 1000 kcal 補充 14g

> **蛋白質雙方法設計**：純熱量比例法在低 TDEE 使用者（老年人、久坐者）可能低估蛋白質需求，故採 `max(熱量比例法, 體重基準法)`。

## 階段四：食物推薦子系統設計（SUB.md）

### 4.1 數學模型

缺口向量、食物填補貢獻、達成率定義，以及兩階段貪婪演算法。

### 4.2 演算法終止條件

```
A：所有 R_j ≥ 90%
B：已選食物種數 = 8
C：剩餘缺口改善幅度 < 1%
```

### 4.3 Phase 3 移除決策

原設計包含 Phase 3（份量微調 ±0.5份迭代），評估後移除。理由：日常飲食本就存在誤差，過度精確的微調對普通使用者沒有實際意義。

## 階段八：完整前端實作

### 8.1 六角形圖表（hexagon.js）

從頭建立，不依賴任何現有 repo。主要開發歷程：

**v1**：基本三層架構（灰色目標環、綠色攝取、藍色食物貢獻）。
**v2**：修正 `canvas.width` vs 邏輯像素 bug（clearRect 使用物理像素導致繪製錯誤）。
**v3**：改用 `setTransform(dpr, 0, 0, dpr, 0, 0)` 取代累積 `scale()`，確保 resize 時不累積變換。
**v4**：加入橘色超出層（evenodd clip 方式，後改為從圓心繪製 excess ratio 多邊形）。
**v5**：加入頂點公克數標籤（白底圓角小標籤）、橘色超出頂點標籤（+Xg）。
**v6**：加入不足虛線（攝取頂點 → 100% 目標點，紅色 dash）。

**初始化 Bug 修正**：`_resize()` 在 layout 完成前被呼叫時 `clientWidth` 可能為 0，導致 size 為負數。改用 `requestAnimationFrame` 延遲至 layout 完成後再計算。

### 8.2 食物貢獻比例計算 Bug

**Bug**：點擊食物卡片，六角圖藍色多邊形顯示極小甚至不可見。

**根本原因**：`foodContribRatios` 函式中多除了一次 `/100`：

```javascript
// 錯誤（使比例縮小 100 倍）
(food.carb * 1.5 / 100) / t.carb_g

// 正確
(food.carb * 1.5) / t.carb_g
```

食物資料庫數值本身即為「每 100g 含量」，150g 用量應直接乘以 1.5，不再除以 100。

### 8.3 攝取量計算 Bug

**Bug**：加入御釜白飯（carb 28.2g/100g），缺口面板顯示碳水增加 42.3g 而非預期 28.2g。

**根本原因**：`eatenFoods` 預設 `amount_g: 150`（1.5 份），計算時 28.2 × 1.5 = 42.3。

**修正**：將預設份量改為 `amount_g: 100`（1 份 = 100g），與資料庫基礎單位一致。同步將 `foodContribRatios` 的乘數從 1.5 改為 1.0。

## 階段九：UI 功能迭代

### 9.1 食物資料庫卡片軸

水平捲動卡片軸，顯示全部食物。分類按鈕過濾，滾輪橫向捲動，箭頭按鈕跳 480px。

**主導營養素徽章**：比較每種食物的碳水/蛋白質/脂肪含量，標示最高者（橘色碳水 / 綠色蛋白質 / 紅色脂肪）。

### 9.2 新增食物功能

可展開表單填寫 8 個欄位，POST `/api/foods` 持久化寫入 `food_db.json`，即時更新卡片軸，id 自動遞增。

### 9.3 今日營養缺口面板

預設顯示，無需計算即可看到佔位說明。加入/移除食物後自動更新進度條與剩餘公克數。

**超標藍字**：超過目標時，右側文字改為藍色（`#1565c0`）。

### 9.4 已加入食物卡片

顯示已加入的食物清單，含食物名稱、分類（100g）、四色营養素標籤，× 按鈕移除。

### 9.5 推薦流程

- 「計算熱量」按鈕：計算後顯示 BMR / TDEE（分兩行），清空所有已加入食物，按鈕改為「重新計算」
- 「取得推薦」：僅在有已加入食物且非全達標時可點擊；全達標時按鈕顯示「已達標」並禁用
- 「取消推薦」：清空推薦清單，六角圖恢復為已加入食物的攝取狀態（不歸零）
- 移除食物時推薦清單自動清空，須重新點擊取得推薦

### 9.6 六角圖視覺化設計演進

最終設計：
- **綠色**：攝取量（clamp 至 100%）
- **橘色**：超出量（`ratio - 1.0`，從圓心繪製，疊在綠色上面）
- **藍色**：食物貢獻預覽（hover 暫時 / click 釘選）
- **紅色虛線**：不足缺口提示
- **頂點白底標籤**：顯示攝取公克數
- **橘色頂點標籤**：超出量（+Xg）
- **目標公克 Grid**：六角圖下方 3×2 表格，計算後顯示

### 9.7 字型縮放歷程

1. 初始字型正常大小
2. 使用者要求「放大 2 倍」→ 所有 CSS font-size × 2，hexagon.js 字型 × 2
3. 使用者要求「變 0.75 倍」→ 加入 `html { zoom: 0.75; }` 全局縮放
4. 最終結果：2 × 0.75 = 1.5 倍原始大小

### 9.8 運動行版面

移除 MET 數字標籤，只顯示下拉名稱，時數輸入，加入橘色 kcal 消耗徽章（`≈ MET × W × 時數 kcal`，即時更新）。

## 階段十：WSL 環境問題

### 10.1 Flask 未安裝

WSL（Debian）使用 `pip install flask` 時出現 `externally-managed-environment` 錯誤（PEP 668）。

**解決方案**：建立虛擬環境：
```bash
python3 -m venv .venv
source .venv/bin/activate
pip install flask
```

### 10.2 中文亂碼

Windows 原生終端預設非 UTF-8 編碼，JSON 中文字顯示亂碼。

**解決方案**：啟動時加 `-X utf8` 旗標：`python -X utf8 nutrition/server.py`
